# 📌 AGENTS.md

## 阅读须知
- 指令优先级：用户显式指令高于本文件，本文件高于默认行为
- 本文件作为全局默认规则，适用于未提供更具体 `AGENTS.md` 的项目或目录；子目录存在 `AGENTS.md` 时，以子目录文件为准
- 工作过程应遵循强制优先、结果导向、可审计原则，确保关键决策与操作可追溯
- 每次回复应在开头提供 `前置说明`；存在外部调用时，应在末尾补充 `工具调用简报`
- 用户显式指令与本文件冲突时，应优先执行用户指令，并在 `前置说明` 中说明偏差原因

## 核心原则
- 优先复用主流稳定库、官方 SDK 与既有能力，默认选择最新稳定版本
- 所有判断应基于完整上下文与可核验证据，不得脱离事实作出结论
- 关键决策、依据、变更与结论应记录至指定位置，保证过程可追溯
- 以结果正确和可验证为先，再考虑实现形式与扩展性
- 标准流程：
**需求理解 → 方案设计 → 实施开发 → 测试验证 → 交付说明 → 部署发布支持**

## 工作语言与输出风格
- 回答、分析、解释与命名说明默认使用中文，优先采用中文术语、表达方式与命名规范
- 思考与推理默认使用中文
- 输出应简洁、直接、结构清晰，避免重复描述
- 非用户明确要求，不主动产出过长文档

## 文档风格
- 需求类文档默认采用 EARS 描述
- 大型文档应分批读取或写入，避免命令执行失败

## 技术栈约束
- 无明确约束时，优先推荐 C# 相关技术栈和工具链
- 新建项目在无明确约束时，默认使用 C# 与 .NET 8.0
- 对既有项目进行设计或修改时，应先说明当前技术栈与工具链；用户无异议时，默认沿用现状
- 设计方案应说明可行选项、当前与未来的优缺点，并给出推荐工具链与技术方案供用户决策

## 文件/目录/编码约束
- 默认目录约定：生产代码位于 `src/`，测试代码位于 `test/`，文档位于 `docs/`，计划文件位于 `plan/`
- 文件编码默认使用 UTF-8 无 BOM，换行默认使用 CRLF
- 代码命名默认使用大驼峰命名；修改或新增文件时，应优先遵循现有风格
- 目录结构在无既有约束时，优先按`业务领域`组织，并使用 `Interfaces/`、`Config/`、`Impl/` 等区分接口、配置与实现

## 注释约束
- 生产代码中的类、接口、方法、属性、字段、事件、枚举、结构体等公开成员或关键成员应补充注释
- 注释内容应说明职责、语义、约束、边界条件、状态变化及与相关类型的关系
- 接口实现类在未扩展、未变更接口语义时，可不重复编写同类注释
- 涉及状态流转、生命周期、并发控制、资源释放、异常约束或边界处理的实现，应在关键位置补充注释
- 测试代码中的类和方法应说明测试目标、场景边界与预期结果；关键断言应标注判定依据
- 修改代码时应保留现有准确注释，仅修补缺失、失真、乱码或与当前语义不一致的部分
- 注释应采用客观陈述句，仅描述职责、语义、关系、状态、约束与边界
- 注释应优先使用定义式、职责式、名词化表达，避免口语化、解释式、推演式和教学式表述
- 注释不得重复代码字面含义，不得包含低信息量或无实际约束价值的内容
- 宜使用以下句式：`表示……`、`用于……`、`负责……`、`对应……`、`提供……`、`维护……`、`记录……`、`作为……入口`、`作为……标记`

## 工具指令约束
- 在 VS Code 开发环境中，本地文件读取、写入、重命名等操作优先使用 `vscode-mcp-server`
- GitHub 仓库、代码片段、Issue、Pull Request 等查询优先使用 `github-mcp-server`
- 优先工具不可用、目标路径不在工作区、权限不足或执行失败时，可回退至其他安全且可审计的方式
- 本地上下文不足时，可补充使用其他检索方式
- 涉及时效性信息、版本信息、在线文档或外部规则时，应先核实后回答，不得仅凭记忆输出
- 严禁 `rm -rf` 等破坏性操作或泄露密钥、令牌、内部链接及其他敏感信息

## Git 约束
- 不得覆盖、回滚或清理未经用户授权的改动
- 不得执行破坏性 Git 操作，包括 `reset --hard`、强制覆盖及其他不可逆操作
- 仓库存在脏工作区时，应先识别变更归属与任务相关性，再决定是否处理
- 提交信息格式统一为 `<type>: <subject>`
- `subject` 简洁明确，概述本次任务变更目标
- `type` 允许值包括：`plan`、`docs`、`test`、`feat`、`fix`、`chore`、`style`、`comment`、`refactor`、`perf`、`revert`、`merge`、`build`、`ci`、`github`

### 提交类型说明
- `plan`：计划、实施方案、任务拆解等计划类文档
- `docs`：说明文档、README、设计说明等文档内容
- `test`：测试用例、测试工程、测试配置与测试修复
- `feat`：新功能开发
- `fix`：缺陷修复
- `chore`：不影响功能的杂项维护
- `style`：格式、排版、规范类调整，例如 `.editorconfig`
- `comment`：仅注释变更
- `refactor`：不改变外部行为的结构调整
- `perf`：性能优化
- `revert`：提交回滚
- `merge`：分支合并
- `build`：构建、编译、依赖、工程配置相关变更
- `ci`：CI/CD 流程相关变更
- `github`：GitHub 工作流、Issue、PR、Actions 等相关变更

## 文档约束
- 文档、图片及 README 相关资源默认存放于 `docs/`，并按资源类型继续划分目录
- README 应聚焦项目定位、当前状态与核心特点，不展开实现细节
- 设计与实现细节统一放在 `Arch.md` 或后续计划、设计文档中
- 代码设计与 README 描述不一致时，应同步修订 README
- 描述原理、流程、结构或交互关系时，应优先补充 Mermaid 图表；图表类型按内容选择

## 代码修改约束
- 涉及破坏性改动时，应先在 `前置说明` 中说明影响范围并征得用户确认
- 获得确认后，应同步清理或修正受影响的代码、接口、配置与文档，保持内容一致

## 推荐工作方式: 任务分级
- 未明确说明时，默认按 L1 执行；满足多个层级时按最高层级处理
- 任务分级用于确定流程强度，工作流用于确定实施方式
- `OpenSpec` 与 `OpenSpec-CN` 视为同类变更工作流；环境中可用哪个即使用哪个
- `OpenSpec` 负责需求、设计、任务、留痕与归档等变更管理；`superpowers` 负责探索、计划、调试、测试、评审与验证等执行方法
- `OpenSpec` 与 `superpowers` 可叠加使用；命中工作流触发条件时，触发条件优先于默认等级

### OpenSpec 触发条件
- 需求不稳定或存在范围讨论
- 需要 proposal、design、tasks、archive 等变更产物
- 需要跨会话推进、多人协作或长期跟踪
- 涉及跨模块、架构级、全局性改动

### superpowers 触发条件
- 需要先探索、写计划、系统化调试、代码评审或高风险验证
- 当前任务可以在单次或少量会话中闭环，但需要更强的方法约束
- 未命中敏捷开发约束时，可按需启用测试驱动开发与完成前验证类技能

### 敏捷开发约束
- 用户明确要求敏捷开发、快速迭代、最小流程或原型验证时，敏捷约束优先于默认 `superpowers` 触发规则
- 敏捷开发场景下，不默认启用测试驱动开发、完整验证、正式评审等重流程型 `superpowers` 技能
- 敏捷开发场景下，`superpowers` 仅在探索、调试、计划、评审或高风险验证确有必要时按需启用
- 最小验证应与改动范围对应；代码改动默认采用编译验证，非代码改动采用回读、静态检查或人工确认
- 涉及公共接口、核心流程、状态流转、边界条件、高回归风险或用户明确要求质量保障时，应补充定向测试或更高强度验证
- 用户明确要求仅保留最小验证时，交付说明应注明验证范围、未覆盖项与残留风险

### L0：轻任务
适用：
- 小范围 Bug 修复
- 文案、样式、配置微调
- 单文件或局部函数改动
- 已有模式下的小幅增量开发

要求：
- 默认直接执行，不强制使用 `OpenSpec` 或 `superpowers`
- 可直接读取、修改、验证
- 不强制产出过程文件
- 交付时说明改动与验证结果即可

### L1：标准任务
适用：
- 多文件联动
- 中等复杂度功能开发
- 页面或接口的局部新增/改造
- 局部重构
- 需要查项目现有实现或外部文档

要求：
- 默认直接执行；命中明确方法型场景时按需使用 `superpowers`
- 做最小充分的上下文收集
- 有清晰实现步骤与验收方式
- 按需生成 `plan/` 留痕文件
- 完成后做明确验证，并说明风险

### L2：重任务
适用：
- 跨模块/架构级改动
- 新业务模块或完整功能链路
- 数据库、权限、部署、核心流程调整
- 需求不清、方案分支多、业务边界复杂

要求：
- 默认使用 `OpenSpec` 或 `OpenSpec-CN` 管理需求、设计、任务与留痕；实施阶段按需叠加 `superpowers`
- 主动引导用户，执行完整的`需求梳理 → 方案设计 → 实施 → 验证 → 交付`流程
- 按需生成 `plan/` 留痕文件
- 明确记录关键决策、风险、验证结论与遗留问题

### L3：全局任务
适用：
- 项目初期，根据需求规划全局架构/技术栈选择

要求：
- 主动按模块化，分层式，依赖注入式，DDD领域驱动式的设计推荐
- 必须使用 `OpenSpec` 或 `OpenSpec-CN` 进行全局规划与分阶段推进；复杂实施环节按需叠加 `superpowers`
- 与`L2：重任务`同等要求

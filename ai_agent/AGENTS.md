# 📌 AGENTS.md

## 阅读须知
- 指令优先级默认用户最高，此文档次之
- 适用于仓库全部目录，若子目录有 `AGENTS.md` 则覆盖
- 坚持“强制优先、结果导向、可审计”，所有流程需可追溯
- 每次回复开头提供`前置说明`，有外部调用时需在末尾补充`工具调用简报`
- 若与本指南冲突的用户显式指令出现，必须遵循并在`前置说明`偏差原因

## 核心原则
- 标准化生态复用优先：推荐时应当优先考虑复用主流稳定库与官方 SDK，锁定最新稳定版本
- 全景分析：结合完整代码上下文与多方证据做出判断，所有结论需有证据支撑
- 透明记录：关键决策、证据、变更需保存在指定目录并可追溯
- 结果导向: 先做对，再做漂亮；先可验证，再谈扩展性
- 标准闭环：
**需求理解 → 方案设计 → 实施开发 → 测试验证 → 交付说明 → 部署发布支持**

## 工作语言与输出风格
- 只允许使用中文回答 - 所有思考、分析、解释和回答都必须使用中文
- 中文优先 - 优先使用中文术语、表达方式和命名规范
- 中文思维 - 思考过程和逻辑分析都使用中文进行
- 输出优先简洁、直接、结构清晰，避免重复描述
- 除非用户明确要求，否则不要写过度冗长的文档

## 技术栈约束
- 若用户无明确约束，默认语言使用C# net8.0
- 设计代码时若无明确说明应当询问用户实现方案使用哪些工具和技术

## 文件/目录/编码约束
- 若无特殊说明生产项目放置于src/,测试项目在test/，文档在docs/，计划文档在plan/
- 默认使用 UTF-8 无BOM编码
- 默认使用 CRLF 换行
- 代码命名默认使用大驼峰命名
- 修改已有文件或新增时优先遵循现有风格
- 目录若无明确风格，优先按`业务目录`为根，Interfaces/Config/Impl区分接口/配置/实现

## 注释约束
- 生产代码中属性、字段、事件、枚举、结构体等需要注释设计含义，类、接口、方法需要注释且说明设计职责，与相关类的关系，接口实现类无更改接口语义则不需要再注释，方法内若有状态流转，边界条件等需要注释
- 测试代码中类和方法需要注释职责，边界条件，测试目标等重要信息，断言需要注释预期结果
- 更改代码时保留现有准确注释，只修补缺失、乱码、失真或与当前语义不一致的部分
- 不写低信息量注释，不重复代码字面内容，不写无意义内容

## 工具指令约束
- 若当前处于vscode开发环境，文件操作如读写文档优先使用`vscode-mcp-server`
- 搜索 GitHub 仓库、代码片段、Issue、PR 优先使用 `github-mcp-server`
- 在本地上下文不足时，补充其他搜索方式
- 涉及最新信息时必须先确认，不要凭记忆回答
- 严禁 `rm -rf` 等破坏性操作或泄露密钥、令牌、内部链接

## Git 约束
- 不要覆盖或回滚用户未授权的改动。
- 不要执行破坏性 Git 操作，如 `reset --hard`、强制覆盖等。
- 若仓库存在脏文件，先识别是否与当前任务相关，再决定是否处理。
- `commit` 提交信息格式为: `<type>: <subject>`
- `commit` subject部分应当应简洁明确，描述本轮变更目标，包括：
  - `plan`：表示计划文档相关，例如 `XX实施计划.md`
  - `docs`：表示说明文档相关，例如 `ReadMe.md`
  - `test`：表示测试相关，例如 `MSTest` / `XUnit`
  - `feat`：表示代码新功能
  - `fix`：表示代码问题修复
  - `chore`：表示一些无关代码的小改动
  - `style`：表示项目规范的变更，例如 `.editconfig`
  - `comment`：表示专门的一次代码注释改动
  - `refactor`：表示代码出现重构
  - `perf`：表示代码优化
  - `revert`：表示提交回滚
  - `merge`：表示代码合并
  - `build`：表示构建相关，例如 `Directory.Build.props`
  - `ci`：表示 `CI/CD` 流程相关
  - `github`：表示 `GitHub` 相关流程

## 推荐工作方式: 任务分级
### L0：轻任务
适用：
- 小范围 Bug 修复
- 文案、样式、配置微调
- 单文件或局部函数改动
- 已有模式下的小幅增量开发

要求：
- 可直接读取、修改、验证
- 不强制产出过程文件
- 交付时说明改动与验证结果即可

### L1：标准任务
适用：
- 多文件联动
- 中等复杂度功能开发
- 页面或接口的局部新增/改造
- 局部重构
- 需要查项目现有实现或外部文档

要求：
- 做最小充分的上下文收集
- 有清晰实现步骤与验收方式
- 按需生成 `plan/L1/` 留痕文件
- 完成后做明确验证，并说明风险

### L2：重任务
适用：
- 跨模块/架构级改动
- 新业务模块或完整功能链路
- 数据库、权限、部署、核心流程调整
- 需求不清、方案分支多、业务边界复杂

要求：
- 主动调用`superpowers skill`或`superpowers-zh skill`
- 主动引导用户, 执行完整的`需求梳理 → 方案设计 → 实施 → 验证 → 交付`流程
- 按需生成 `plan/L2` 留痕文件
- 明确记录关键决策、风险、验证结论与遗留问题

### L3：全局任务
适用：
- 项目初期，根据需求规划全局架构/技术栈选择

要求：
- 主动按模块化，分层式，依赖注入式的设计推荐
- 与`L2：重任务`同等要求，但按需生成`plan/` 留痕文件

# 创建实施计划的风格提示词

## 指令优先级

1. 若与其他文档或用户在当前对话中的明确要求冲突则需要询问用户意见

## 计划与实施规则

### 1. 轮次工作先写计划

- 如果当前是没有计划文档，应当询问是否整理整体业务设计和技术架构的设计，整理到根目录 `plan/Target/...`，并进行文件拆分，在`plan/Target/蓝图入口.md`中进行全局说明和文件引用，包括但不限于`技术选择方案/业务构建方案/系统架构方案/数据流说明/实施路径说明等`被拆分出并被`蓝图入口.md`引用的文件
- 如果该轮涉及新设计、新蓝图或主语义调整，应当先在`plan/Target/蓝图入口.md`中补充说明，再补本轮设计 `plan/Target/...`，再写本轮实施计划 `plan/Execute/...`
- 不直接跳过计划进入实现

### 2. 默认执行节奏

- 默认按“逐个任务执行 + 每个任务完成即单独中文 commit”的节奏推进
- 默认使用 Inline Execution，不使用多智能体 / 子 agent，除非用户明确要求
- 一个任务没有验证和提交前，提示用户当前情况，不进入下一个任务

### 3. 计划文档固定要求

每一轮实施计划默认保持现有仓库风格，通常应包含：

- `目标: 表示此计划的目标，并说明目标应该如何设计，按照目标拆分后有哪些任务`
- `架构: 表示计划对业务架构设计带来的影响，并承前启后说明对前一轮和后一轮的影响`
- `技术栈: 计划涉及的技术栈`
- `影响范围: 表示影响到的业务范围，例如删除XXX类，更改XXX枚举，新增XXX功能，XXX功能变更为XXX`
- `设计理解: 对新增的语义使用大白话进行说明`
- `任务说明: 按照目标中的任务排序且拆分为步骤，每个任务最后一条步骤都应该是提交任务，按照当前文档的输出与提交要求提交commit信息`
- `自检清单: 需要进行测试的单元清单`
- `执行建议: 对任务的执行建议`
- `后续计划: 对后续实施计划的展望和说明`

每一轮计划末尾都要补 `后续计划`，至少说明下一轮和下两轮的暂定目标。

### 4. 文档同步

当代码实现改变了主语义、契约、示例或运行方式时，必须同步更新：

- `README.md`
- 对应 `plan/Target/...`
- 对应 `plan/Execute/...`
- 必要时更新 `plan/Target/蓝图入口.md`

## 注释规则

### 1. 常规代码改动

- 新增或修改代码时，要同步检查并补齐相关注释
- 不允许继续新增“无注释关键类型”“口水化注释”“只复述符号名的注释”

### 2. 纯注释任务

如果当前任务是“只整理注释”：

- 只改注释
- 不改逻辑、命名、结构、接口签名和断言
- 不运行 `build` 或 `test`

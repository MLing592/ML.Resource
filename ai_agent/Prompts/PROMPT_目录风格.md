# 目录风格提示词

## 用途

用于约束本解决方案业务模块的目录分层、命名空间划分和继续拆分方式，确保后续新增或重构模块时，目录结构、职责边界和命名规则保持一致。

## 总原则

- 全部使用中文说明。
- 优先遵循当前项目已经形成的模块分层方式，若与此文档不一致询问按此文档分层还是原有方式。
- 目录结构首先表达职责边界，其次才是技术细节。
- 命名空间与目录保持一一对应，避免目录和命名空间脱节。
- 同一模块内，接口、实现、配置、共享模型要稳定分层，不混放。
- 如果当前业务规则已经较大，则先按业务继续分层，再在该业务层内部继续保持统一的四类目录规则。

## 当前项目的模块分层规则

以单个业务模块为单位，默认采用以下四类目录：

- `Interfaces`
- `Impl`
- `Config`
- `Shared`

对应的命名空间也保持一致：

- `...<模块名>.Interfaces`
- `...<模块名>.Impl`
- `...<模块名>.Config`
- `...<模块名>.Shared`

例如 `GracefulShutdown` 模块应遵循：

```text
GracefulShutdown/
  Interfaces/
  Impl/
  Config/
  Shared/
```

## 各目录职责

### `Interfaces`

用于放模块对外暴露的抽象契约，例如：

- 服务接口
- 协调器接口
- 提供器接口
- 模块内部需要稳定依赖反转的抽象

要求：

- 接口文件放在 `Interfaces`，不要和实现类混放。
- 接口命名保持 `I` 前缀。
- 接口命名空间必须落在 `.Interfaces`。

### `Impl`

用于放接口的默认实现和模块内部主要执行逻辑，例如：

- 服务实现
- 协调器实现
- 处理器实现
- 扫描器、调度器、执行器等具体类

要求：

- 具体实现统一放在 `Impl`，不要散落到模块根目录。
- 实现类命名空间必须落在 `.Impl`。
- 一个实现如果只是某个接口的默认实现，也仍然放在 `Impl`，不要因为文件少而回退到根目录。

### `Config`

用于放模块配置和装配相关类型，例如：

- `Options`
- 配置校验
- 配置映射
- 模块级 `ServiceCollection` 扩展
- 与配置读取、依赖装配直接相关的类型

要求：

- 只要核心职责是“配置”或“装配”，就进入 `Config`。
- `Options`、配置扩展、模块注册入口不要混进 `Impl`。
- 命名空间必须落在 `.Config`。

### `Shared`

用于放模块内跨层共享的通用模型与公共语义，例如：

- 枚举
- 结果模型
- 状态模型
- 业务通用 DTO
- 常量
- 跨 `Interfaces` / `Impl` / `Config` 都会引用的轻量公共类型

要求：

- `Model`、`EnumType`、`Constants` 统一收敛到 `Shared`并建立Model/EnumType/Constants/DTOs等目录，若某一项数量较少只有5项以下则统一到一个文件里，例如GracefulShutdownModel。
- 只要是业务通用模型，优先放 `Shared`，而不是再建 `Model`。
- `Shared` 放的是“跨层共用的业务语义模型”，不是“杂项回收站”。

## 继续分层规则

如果某个模块业务继续膨胀，单层的 `Interfaces / Impl / Config / Shared` 已经过大，则先按业务拆一层，再在业务层内部继续保持同样规则。

例如：

```text
ReliableMessaging/
  Delivery/
    Interfaces/
    Impl/
    Config/
    Shared/
  Subscription/
    Interfaces/
    Impl/
    Config/
    Shared/
  Routing/
    Interfaces/
    Impl/
    Config/
    Shared/
```

也就是说：

- 先按业务域拆目录
- 再在每个业务域内部继续保持 `Interfaces / Impl / Config / Shared`
- 不要一部分业务按职责分层，另一部分业务按文件类型散放

## 不建议的做法

- 不要把接口和实现放在同一目录。
- 不要把 `Options`、模块注册扩展、配置读取逻辑放进 `Impl`。
- 不要把只在单个实现内部使用的纯私有辅助类型塞进 `Shared`。
- 不要为了文件数量少就把目录规则打平到模块根目录。
- 不要出现目录在 `Interfaces`、命名空间却在 `.Impl` 这类不一致情况。

## 生成或重构代码时的执行要求

当需要为解决方案新增模块、整理目录或重构现有模块时，请遵循以下要求：

1. 先判断这是一个独立业务模块，还是某个大模块下的子业务。
2. 独立业务模块默认建立 `Interfaces / Impl / Config / Shared` 四类目录。
3. 如果只是大模块下的一个子业务，则先进入子业务目录，再建立相同四类目录。
4. 接口进 `Interfaces`，实现进 `Impl`，配置与装配进 `Config`，共享模型进 `Shared`。
5. 如果现有模块已经按该风格分层，新增文件时必须沿用，不另起炉灶。
6. 如果原有注释、命名和结构已经符合该风格，优先保持，不做无意义搬动。

## 示例提示词

以后可以直接这样说：

- “按 `docs/prompts/PROMPT_目录风格.md` 的规则整理这个模块目录”
- “把这个新模块按项目的 `Interfaces / Impl / Config / Shared` 风格落目录”
- “按项目目录风格重构 `GracefulShutdown`”
- “如果业务已经太大，先按业务拆层，再按 `Interfaces / Impl / Config / Shared` 细分”

## 注意

- 我不会自动永久记住项目私有的目录分层规则，除非你在当前任务里明确引用这份文档。
- 最有效的方式不是只说“按之前风格”，而是直接引用 `docs/prompts/PROMPT_目录风格.md`。

# 注释风格提示词

## 用途

用于约束本仓库中测试代码与生产代码的注释风格，确保后续新增或修改代码时，注释语言、粒度和表达方式保持一致。

## 总原则

- 全部使用中文。
- 代码文件优先保持原编码，例如原本是UTF8-BOM则继续保持UTF8-BOM。
- 表达要简洁、直接、工程化。
- 注释只解释职责、边界、约束、关键流程和非显然原因，不重复代码字面内容。
- 不写鼓励性语言，不写“这里是为了……”“这个地方……”这类低信息量注释。
- 注释整理任务中，只允许修改注释，不修改逻辑、命名、结构和断言，也不运行 `build` 或 `test`。

## 测试代码注释风格

### 规则

```md
请按以下固定风格为测试代码补充或整理注释，只允许修改注释，不要修改测试逻辑、断言、命名和结构，也不要运行测试。

要求如下：

1. 所有测试类都必须添加中文 XML 注释。
2. 测试类注释必须说明：
   - 该测试类验证的测试范围
   - 整体预期结果
3. 所有测试方法都必须添加中文 XML 注释。
4. 测试方法注释必须明确写出：
   - 测试目标
   - 预期结果
5. 在每组 `Assert` 断言之前，必须添加中文单行注释，格式统一为：
   - `// 预期：......`
6. 注释风格要求：
   - 全部使用中文
   - 简洁、直接、工程化
   - 不写低信息量注释
7. `Assert` 前的“预期”注释必须满足：
   - 直接描述该组断言要验证的结果
   - 不重复代码字面内容
   - 不写空话
   - 一组语义一致的断言前写一条“预期”注释即可
8. 不需要做的事情：
   - 不修改代码
   - 不调整测试逻辑
   - 不新增测试
   - 不运行 build 或 test

如果发现某个测试文件已经有注释，则按上述风格统一整理，而不是保留混杂风格。
```

## 生产代码注释风格

### 规则

```md
为代码补充或整理注释，只允许修改注释，不要修改逻辑、命名、结构和接口签名，也不要运行 build 或 test。

要求如下：

1. 类型、接口和公开成员使用中文 XML 注释。
2. 注释必须简洁、直接、工程化。
3. 复杂私有方法补中文 XML 注释；关键分支可补少量中文行内注释。
4. 重点说明职责、调用时机、边界条件、状态流转和非显然约束等带有流程推进信息的内容。
5. 不写低信息量注释，不重复代码字面内容。
6. 不使用测试专用的 `// 预期：` 写法。
7. 如果发现注释乱码或内容已经过时，要按当前实现修正。
8. 保留现有准确注释，只修补缺失、乱码、失真或与当前实现不一致的部分。

不需要做的事情：
- 不修改业务逻辑
- 不调整代码结构
- 不变更对外 API
- 不运行 build 或 test
```
